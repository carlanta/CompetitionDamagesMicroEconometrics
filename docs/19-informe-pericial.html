19  Redacción y defensa del informe pericial

19.1 Del modelo al documento

Un informe pericial de daños no es un artículo académico. No es un manual de econometría. No es un ejercicio de lucimiento técnico. Es un documento probatorio cuyo único propósito es persuadir a un tribunal de que la cuantificación del daño es correcta, razonable y está debidamente fundamentada.

He presenciado a lo largo de mi carrera numerosos casos en los que un análisis técnicamente impecable fracasó procesalmente porque el informe era incomprensible, desestructurado o incapaz de responder a las preguntas que el tribunal necesitaba resolver. Inversamente, he visto informes más sencillos que resultaron devastadoramente eficaces porque estaban bien escritos, bien organizados y bien defendidos.

Este capítulo recoge las lecciones de esa experiencia.

19.2 Estructura del informe

Un informe pericial de daños debe contener, como mínimo, las siguientes secciones:

Code 
tibble(
  Sección = c("1. Resumen ejecutivo",
              "2. Mandato y cualificación del perito",
              "3. Hechos relevantes",
              "4. Metodología",
              "5. Datos utilizados",
              "6. Resultados",
              "7. Análisis de sensibilidad",
              "8. Conclusiones",
              "Anexos"),
  Contenido = c("Resultado principal en 1-2 páginas. El juez puede no leer más",
                 "Qué se le pide al perito y por qué está cualificado",
                 "Descripción objetiva de la conducta y el mercado afectado",
                 "Método elegido, justificación, supuestos, limitaciones",
                 "Fuentes, período, limpieza, variables construidas",
                 "Estimación central con tablas y gráficos",
                 "Escenarios, tornado chart, robustez",
                 "Cuantía final, descomposición, fecha de referencia",
                 "Datos brutos, código R, tablas detalladas, CV del perito"),
  `Público principal` = c("Juez", "Juez", "Juez / abogados",
                           "Perito contrario", "Perito contrario",
                           "Juez / abogados", "Juez / perito contrario",
                           "Juez", "Técnico")
) |>
  knitr::kable(format = "pipe", align = c("l", "l", "l"))

La Table 19.1 refleja un principio que considero fundamental: el informe tiene múltiples lectores con niveles de conocimiento técnico muy distintos. El resumen ejecutivo es para el juez. La metodología es para el perito contrario. Los anexos son para quien quiera verificar los cálculos.

19.3 Cómo presentar resultados econométricos a no economistas

La regla de oro es: el resultado debe entenderse sin la ecuación. El juez no necesita ver \(\hat{\beta} = 0.157\) para entender que el cártel elevó los precios un 17%. Lo que necesita es una frase clara, una tabla limpia y un gráfico que muestre la brecha entre lo observado y el contrafactual.

Algunas pautas prácticas:

  • Traducir coeficientes en magnitudes comprensibles: porcentajes, euros, meses.
  • Usar gráficos con dos líneas (observado vs. contrafactual) en lugar de tablas de regresión.
  • Reservar las tablas de regresión para los anexos técnicos.
  • Incluir intervalos de confianza como rangos, no como errores estándar.
  • Explicar el análisis de sensibilidad como “el resultado se mantiene entre X e Y bajo supuestos razonables”.

19.4 Tablas y gráficos para informes judiciales

Code 
tibble(
  Concepto = c("Sobreprecio estimado", "Período cartelizado",
               "Cantidad afectada (acumulada)", "Daño directo total",
               "Intereses compensatorios (4%)", "Daño total actualizado"),
  Valor = c("17.0%", "Ene 2021 – Dic 2022 (24 meses)",
            "15.200 unidades", "488.558 EUR",
            "52.340 EUR", "540.898 EUR")
) |>
  knitr::kable(format = "pipe", align = c("l", "r"))

La Table 19.2 muestra el tipo de tabla que un tribunal espera ver: cifras claras, unidades explícitas, descomposición comprensible. No hay notación econométrica. No hay p-valores. No hay jerga. Solo la respuesta a la pregunta que importa: ¿cuánto vale el daño?

19.5 La comparecencia: defender el informe ante el tribunal

La defensa oral del informe pericial es, en muchos procedimientos, el momento decisivo. El perito debe estar preparado para:

  1. Explicar el método en 5 minutos, sin jerga, con un gráfico de apoyo.
  2. Responder a las objeciones del perito contrario con calma, datos y argumentos económicos.
  3. Reconocer las limitaciones de su análisis sin que eso socave la credibilidad del resultado.
  4. No improvisar: si no tiene la respuesta, decirlo y ofrecer proporcionarla por escrito.
  5. Mantener la objetividad: el perito no es abogado del cliente. Su credibilidad depende de que el tribunal lo perciba como independiente.

19.6 Reproducibilidad

Todo informe pericial moderno debería ser reproducible. Esto significa que cualquier persona con acceso a los datos brutos y al código debería poder generar exactamente las mismas tablas, gráficos y resultados que aparecen en el informe.

Code 
# Estructura de un proyecto reproducible
# proyecto/
# ├── data/          ← datos brutos (nunca modificar)
# ├── scripts/       ← código R para cada capítulo/sección
# ├── output/        ← gráficos y tablas generados
# ├── informe.qmd    ← documento principal (Quarto/Rmarkdown)
# └── README.md      ← instrucciones de ejecución

# Para regenerar el informe completo:
# quarto render informe.qmd

La reproducibilidad no es un capricho académico: es una salvaguarda procesal. Cuando la contraparte solicita los datos y el código —y cada vez lo hace con más frecuencia—, el perito que puede entregar un repositorio limpio y ejecutable se encuentra en una posición muy superior al que trabaja con hojas de cálculo opacas.

19.7 Lo que un tribunal debe retener

Un buen informe pericial de daños es aquel que responde a la pregunta «¿cuánto vale el daño?» de forma clara, fundamentada y verificable. La calidad técnica es condición necesaria pero no suficiente: el informe debe ser comprensible para el juez, resistente al escrutinio del perito contrario y reproducible por cualquier tercero que acceda a los datos y al código.

Este manual ha pretendido proporcionar las herramientas para conseguirlo. Si el lector ha llegado hasta aquí, dispone de los fundamentos conceptuales, los métodos econométricos, las implementaciones en R y las pautas prácticas para cuantificar daños antitrust con rigor y comunicarlos con eficacia. El resto depende de la experiencia, del buen juicio y de una honestidad profesional que ningún manual puede enseñar.
